## 变更内容

### 1. 新增 IParsable 接口

- 在实现方案中新增了 `IParsable` 接口定义
- 使用静态抽象成员 `static abstract object? TryParse(object? obj)`
- 作为类型级别的可分析能力标记

### 2. 更新源生成逻辑

- **TryParse 函数逻辑变更**：
  - 原方案：使用 try-catch 包装 Parse 函数
  - 新方案：先判断目标类型是否实现了 IParsable 接口
    - 有实现 → 直接调用 `IParsable.TryParse` 方法
    - 没有实现 → 使用默认解析逻辑

- **Parse 函数逻辑变更**：
  - 原方案：直接处理类型转换逻辑
  - 新方案：先调用 `TryParse` 函数，再判断结果是否为空

## 变更原因

根据 [需求补充.md](./1.8-需求补充.md) 中的补充内容：
1. 需要添加 `IParsable` 可分析接口
2. 源生成逻辑需要更新，先检测 IParsable 接口再决定解析方式
3. 需要为基础 Model 对象添加独立的 Parse 和 TryParse 函数

## 可能产生的影响

### 正面影响
1. **灵活性提升**：自定义类型可以通过实现 IParsable 接口控制解析逻辑
2. **代码复用**：基础 Model 对象提供统一的解析能力
3. **架构清晰**：解析逻辑职责分离（源生成器检测、类型实现、基础类型处理）

### 潜在风险
1. **.NET Standard 2.0 兼容性**：静态接口成员（static abstract）需要 .NET 7+ 支持
   - TODO: 需要确认项目的目标框架版本
2. **接口变更影响**：IBaseModel 接口新增函数会影响所有实现类
3. **源生成器复杂性**：增加了类型检测和条件生成逻辑
