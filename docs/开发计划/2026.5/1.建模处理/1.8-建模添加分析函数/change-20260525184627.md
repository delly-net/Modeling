## 变更内容

### 1. 新增 .NET Standard 2.0 兼容性约束

- 在功能概述的**重要约束**中新增：
  - `.NET Standard 2.0` 不支持 `Nullable` 类型，接口定义时应使用非空类型 `object` 而非 `object?`
  - 此项为项目默认规范，所有函数定义时都需要默认考虑不同 .NET 版本的兼容性

### 2. 更新接口定义

- **IParsable 接口**：
  - 原方案：`static abstract object? TryParse(object? obj)`
  - 新方案：`static abstract object TryParse(object obj)`

- **IBaseModel 接口**：
  - 原方案：`object Parse(object? obj)` 和 `object? TryParse(object? obj)`
  - 新方案：`object Parse(object obj)` 和 `object TryParse(object obj)`

### 3. 更新源生成逻辑

- **TryParse 函数逻辑变更**：
  - 原方案：无 IParsable 实现时使用默认解析逻辑
  - 新方案：无 IParsable 实现时抛出异常
  - 原因：强制类型实现 IParsable 接口以获得解析能力

```csharp
// 新方案：没有实现 → 抛出异常
throw new ArgumentException($"Type {ClassName} does not implement IParsable interface");
```

## 变更原因

根据 [需求补充.md](./1.8-需求补充.md) 中的补充内容：
1. `.Net Standard 不支持Nullable类型，可定义为 object TryParse(object obj)`
2. 此项为项目默认规范，所有函数定义时都需要默认考虑不同.Net版本的兼容性
3. 源生成逻辑中，没有实现 `IParsable` 时应该抛出异常

## 可能产生的影响

### 正面影响
1. **跨框架兼容性**：确保代码在 .NET Standard 2.0 和 .NET 6.0 上都能正常运行
2. **强制规范**：通过抛出异常强制类型实现 IParsable 接口，避免隐藏的解析行为
3. **代码一致性**：统一使用非空类型 `object`，避免可空类型在不同框架上的差异

### 潜在风险
1. **静态接口成员与 .NET Standard 2.0**：静态接口成员（static abstract）仍需要 .NET 7+ 支持
   - 注意：项目目标框架为 `netstandard2.0` 和 `net6.0`，静态抽象成员在 net6.0 上不可用
   - TODO: 需要确认解决方案（可能需要在 net6.0 上使用条件编译或替代方案）
2. **可空类型语义丢失**：使用非空类型 `object` 后，无法通过编译器检查 null 相关的潜在问题
