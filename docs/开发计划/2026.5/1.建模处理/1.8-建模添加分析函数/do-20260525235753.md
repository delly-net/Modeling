4. 源生成器通过 `ParsableAttribute` 特性和 `IParsable<T>` 泛型参数找到实现类

## 技术约束分析

### 关键发现

1. **历史执行问题**：之前的实现使用反射调用静态方法，违反了"本项目不允许出现反射方式调用"的要求
2. **命名冲突**：.NET 6.0+ 内置的 `System.IParsable<TSelf>` 与我们的 `Delly.Modeling.IParsable<T>` 冲突，需要在生成的代码中使用完全限定名
3. **AOT 限制**：`netstandard2.0` 项目不支持 AOT 编译，但主项目 (.NET 10.0) 运行正常

### 调整方案

1. 移除反射方案，改为编译时生成 `new {ParserClassName}()` 调用
2. 使用 `Delly.Modeling.IParsable<T>` 完全限定名避免命名冲突
3. 源生成器扫描程序集查找 `[Parsable]` 标记的类，验证其 `IParsable<T>` 泛型参数

## 技术决策记录

### 为什么移除反射方案？

1. **约束违反**：Prompt 明确要求"本项目不允许出现反射方式调用"
2. **AOT 兼容性**：反射可能导致 AOT 编译问题
3. **编译时检查**：使用 `new {ParserClassName}()` 可在编译时验证类型存在

### 为什么使用泛型接口而非静态抽象成员？

1. **源生成器限制**：`Delly.Modeling.Generator` 必须在 `netstandard2.0` 上编译
2. **向后兼容性**：泛型接口在 `netstandard2.0` 和 `net6.0` 上均可工作

### 为什么使用完全限定名？

1. **命名冲突**：.NET 6.0+ 内置的 `System.IParsable<TSelf>` 与我们的 `Delly.Modeling.IParsable<T>` 冲突
2. **明确性**：完全限定名使代码意图更清晰

## 使用示例

```csharp
// 创建 IParsable<Order> 的实现类
[Parsable]
public class OrderParser : IParsable<Order>
{
    public object? TryParse(object? obj)
    {
        if (obj == null)
            return null;

        if (obj is Order order)
            return order;

        if (obj is long id)
            return new Order { Id = id, Name = string.Empty };

        return null;
    }
}

[Modelable]
public partial class Order
{
    public long Id { get; set; }
    public string Name { get; set; }
}
```
