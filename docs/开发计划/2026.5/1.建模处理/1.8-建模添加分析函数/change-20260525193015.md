## 变更原因

根据 `需求补充.md` 和 `注意事项.md` 中的项目规范，发现原始功能文档存在以下不符合规范的地方：

1. 原始文档使用了 `static abstract` 在接口定义中，但项目明确禁止在接口中使用静态抽象成员
2. 原始文档使用了可空类型 `object?` 在接口定义中，但根据 `.NET 版本兼容性规范`，接口定义应使用非空类型 `object`
3. 项目不允许使用反射方式调用

## 变更内容

### 1. IParsable 接口设计变更

**原设计**：
```csharp
public interface IParsable
{
#if !NETSTANDARD2_0
    static abstract object? TryParse(object? obj);
#else
    static abstract object TryParse(object obj);
#endif
}
```

**新设计**：
```csharp
public interface IParsable
{
    // 标记接口，无需定义成员
    // 类型应提供静态方法：public static object TryParse(object obj)
}
```

### 2. 接口定义类型变更

**原设计**：使用可空类型 `object?`
**新设计**：使用非空类型 `object`

### 3. 移除条件编译

**原设计**：使用 `#if !NETSTANDARD2_0` 区分不同框架
**新设计**：统一使用非空类型 `object`，实现类内部处理 null 值

### 4. 源生成器调用方式变更

**原设计**：使用反射或条件编译调用静态接口成员
**新设计**：源生成器硬编码调用类型的静态 `TryParse` 方法

## 可能产生的影响

1. **正向影响**：
   - 移除 `static abstract` 后，代码兼容性更好，支持 .NET Standard 2.0
   - 使用标记接口模式更灵活，类型可以自定义静态方法名称（但建议统一使用 `TryParse`）
   - 避免了运行时反射调用，AOT 友好

2. **需要注意的事项**：
   - 类型实现 `IParsable` 接口时，需要自行提供静态 `TryParse` 方法
   - 源生成器生成代码时，需要硬编码调用类型名称和静态方法
   - 需要确认静态方法命名规范是否强制为 `TryParse`
