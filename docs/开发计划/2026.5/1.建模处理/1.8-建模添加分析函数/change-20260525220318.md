## 变更内容

### 1. IParsable 接口定义变更

**原始实现**：
- 非泛型标记接口 `IParsable`，不定义成员
- 类型提供静态 `TryParse` 方法
- 源生成器硬编码调用静态方法

**新实现**（按照需求补充）：
- 泛型接口 `IParsable<T>`，定义 `TryParse` 方法
- 接口成员：`object? TryParse(object? obj)`（非 .NET Standard 2.0）/ `object TryParse(object obj)`（.NET Standard 2.0）
- 源生成器实例化实现对象并调用实例方法

### 2. 源生成逻辑变更

**原始实现**：
```csharp
if (typeof({ClassName}).IsAssignableTo(typeof(IParsable)))
{
    return {ClassName}.TryParse(obj);  // 静态方法调用
}
```

**新实现**（按照需求补充）：
```csharp
// 查找 IParsable<{ClassName}> 的实现类型
// 通过 new {实现类型名}() 实例化对象
// 调用实例对象的 TryParse 方法
```

### 3. 用户代码约束变更

**原始约束**：
- 类型实现 `IParsable` 标记接口
- 提供静态 `TryParse` 方法

**新约束**：
- 类型需要提供 `IParsable<{ClassName}>` 的实现类
- 实现类提供无参构造函数
- 实现类的 `TryParse` 方法返回解析后的 `{ClassName}` 实例

## 变更原因

需求补充文档要求使用泛型接口 `IParsable<T>`，并实例化对象调用实例方法，而非使用标记接口 + 静态方法的方式。

## 可能产生的影响

1. **用户代码复杂度增加**：
   - 需要单独创建实现类
   - 实现类命名约定需要明确
   - 需要提供无参构造函数

2. **源生成器复杂度增加**：
   - 需要查找 `IParsable<{ClassName}>` 的实现类
   - 需要处理多个实现类的情况
   - 需要实例化对象（可能产生垃圾）

3. **性能影响**：
   - 每次解析都需要实例化对象（而非静态方法）
   - 增加对象分配和垃圾回收压力
