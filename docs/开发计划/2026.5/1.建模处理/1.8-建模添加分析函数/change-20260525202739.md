## 变更内容

### 1. 源生成器 TryParse 实现方式变更

**原始实现**：使用反射动态查找和调用静态 TryParse 方法
**新实现**：硬编码调用 ClassName.TryParse 方法

**影响文件**：
- `Delly.Modeling.Generator/ModelableSourceGenerator.cs`
- `Delly.Modeling.Generator/ModelTableSourceGenerator.cs`

### 2. 没有实现 IParsable 时的异常类型变更

**原始实现**：返回 null
**新实现**：抛出 NotSupportedException

**原因**：符合禁止反射的约束，明确告知用户需要实现 IParsable 接口

### 3. IParsable 接口注释更新

**原始注释**："源生成器会通过反射检测并调用此方法"
**新注释**："源生成器会检测类型是否实现此接口，并硬编码调用其静态 TryParse 方法"

**原因**：移除反射相关描述，符合新的实现方式

### 4. 异常类型统一

**Parse 函数异常**：保持 ArgumentException（参数无效语义）
**TryParse 函数异常（无 IParsable 实现）**：使用 NotSupportedException（不支持的操作语义）

## 变更原因

1. **禁止反射约束**：需求补充明确要求"本项目不允许出现反射方式调用"
2. **AOT 兼容性**：硬编码调用方式更适合 AOT 编译
3. **明确约束**：通过抛出异常明确告知用户需要实现 IParsable 接口

## 可能产生的影响

1. **用户代码约束**：
   - 用户类型必须实现 IParsable 接口
   - 必须提供正确签名的静态 TryParse 方法
   - 方法签名必须为：`static object? TryParse(object? obj)`（非 .NET Standard 2.0）或 `static object TryParse(object obj)`（.NET Standard 2.0）

2. **编译时检查**：
   - 如果用户类型没有实现 IParsable，运行时会抛出异常
   - 如果用户类型提供了不正确的静态方法签名，会产生编译错误

3. **向后兼容性**：
   - 现有未实现 IParsable 的类型在调用 Parse/TryParse 时会抛出异常
   - 需要用户主动实现 IParsable 接口才能使用解析功能
