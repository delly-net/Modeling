## 变更内容

### 1. TryGetModel 方法签名调整

**原设计**:
```csharp
public bool TryGetModel<T>(out IEntityModel? model) where T : class
{
    model = typeof(T) switch
    {
        var t when t == _userType => User.GetEntityModel(),
        var t when t == _orderType => Order.GetEntityModel(),
        _ => null
    };
    return model is not null;
}
```

**新设计**（来自需求补充）:
```csharp
/// <summary>
/// 尝试获取指定类型的模型
/// </summary>
/// <typeparam name="T">模型类型</typeparam>
/// <returns>模型对象，未找到时返回 null</returns>
public IEntityModel? TryGetModel<T>() where T : class
{
    return typeof(T) switch
    {
        var t when t == _userType => User.GetEntityModel(),
        var t when t == _orderType => Order.GetEntityModel(),
        _ => null
    };
}
```

### 2. 明确泛型方法数量

**原设计**: TODO 项中询问是否需要同时支持多种方法

**新需求**: 明确只需要以下两个方法，不需要额外添加其他方法
- `IEntityModel GetModel<T>() where T : class`
- `IEntityModel? TryGetModel<T>() where T : class`

### 3. 核心差异点

| 项目 | 原设计 | 新设计 |
|:---|:---|:---|
| TryGetModel 返回类型 | `bool` | `IEntityModel?` |
| TryGetModel 参数 | `out IEntityModel? model` | 无参数 |
| 泛型方法数量 | 待确认 | 确认为 2 个 |
| Nullable 考虑 | 未提及 | 需要考虑不同 .NET 版本的兼容性 |

## 变更原因

根据需求补充文档反馈：
1. 泛型方式获取建模，只有以下两个，不需要额外添加其他方法
2. 需要考虑不同 .NET 版本的 nullable 兼容性

## 可能产生的影响

1. **API 简化**: `TryGetModel<T>()` 使用更简洁，不需要声明 out 参数
2. **Nullable 兼容性**: 需要在源生成器中处理不同 .NET 版本的 nullable 语法
   - .NET Standard 2.0: 使用 `#if !NETSTANDARD2_0` 包裹 nullable 代码
   - .NET 6.0+: 可直接使用 `?` 语法
3. **方法一致性**: 两个方法签名更加对称，都是返回值方式
