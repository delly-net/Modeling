# 1.16 需求补充

## 需求完善

本文档用于补充任务 1.16 执行过程中的需求细节和规范说明。

### 补充说明

1. **修改范围明确**：
   - 仅修改 `Delly.Modeling` 和 `Delly.Modeling.Generator` 两个项目的代码文件
   - `Deme` 和 `NugetDemo` 项目已经配置了 `<Nullable>enable</Nullable>`，无需修改

2. **源生成器代码模板特殊处理**：
   - 源生成器代码文件本身的 `#nullable enable` 指令需要移除
   - 但源生成器**生成的代码模板**中的 `#nullable enable` 指令必须保留
   - 这确保生成的代码在目标项目中具有正确的 nullable 行为

3. **条件编译保留**：
   - 方法签名中的条件编译 `#if !NETSTANDARD2_0` 必须保留
   - 这是用于区分不同 .NET 版本的接口签名差异

## 规范要求

1. **csproj 配置位置**：
   - `<Nullable>enable</Nullable>` 应放在 `<PropertyGroup>` 内
   - 建议放在目标框架配置之后，便于查看
   - .NET Standard 2.0 不需要配置 `<Nullable>enable</Nullable>`
   - 多框架项目配置 `<Nullable>enable</Nullable>` 时需要加上指定框架兼容性适配条件
   ```
   <!-- 仅给支持的框架开启 Nullable -->
	<PropertyGroup Condition="'$(TargetFramework)'!='netstandard2.0'">
		<Nullable>enable</Nullable>
	</PropertyGroup>

   ```

2. **代码修改一致性**：
   - 移除 nullable 指令时，一并移除相关的空行和注释
   - 保持代码格式整洁

3. **编译验证**：
   - 修改后必须对 `netstandard2.0` 和 `net6.0` 两个目标框架分别进行编译验证
   - 确保两个目标框架都没有编译警告或错误

## 待确认清单

### 已确认项

- [x] .NET Standard 2.0 目标框架编译器会忽略 nullable 配置
- [x] 不需要在 csproj 中使用条件编译区分不同目标框架的 nullable 配置

### 待确认项

- [ ] 源生成器生成的代码是否需要保留 nullable 指令
  - **建议**：保留，确保生成代码的自包含性和正确性
    - 保留

- [ ] 是否需要为不同目标框架设置不同的 nullable 配置
  - **建议**：不需要，统一使用 `<Nullable>enable</Nullable>` 即可
    - 根据目标框架兼容性来设定
