# 变更记录 - 20260523202211

## 原文档关键内容

### 原设计假设

1. **属性元数据方式**：假设 `[MoQuery]` 与 `[MoColumn]` 搭配使用
   - 使用示例展示了查询对象属性使用 `[MoColumn]` 特性
   - 实现方案提到"支持读取 `[MoColumn]` 特性的元数据"
   - 属性访问器生成逻辑复用 `[MoColumn]` 的元数据读取

2. **源生成逻辑**：两种特性使用相同的属性元数据处理
   - `GetPropertyMetadata` 方法读取 `[MoColumn]` 特性
   - 生成相同的 `IEntityModelProperty` 接口实现

### 原使用示例

```csharp
[MoQuery]
public partial class UserQuery
{
    [MoColumn(Name = "username", Type = ColumnType.VARCHAR, Length = 50)]
    public string? Username { get; set; }

    [MoColumn(Name = "min_age", Type = ColumnType.INTEGER)]
    public int? MinAge { get; set; }

    [MoColumn(Name = "max_age", Type = ColumnType.INTEGER)]
    public int? MaxAge { get; set; }
}
```

### 原 IColumnMetadata 接口依赖

`IEntityModelProperty` 接口包含以下需要从 `[MoColumn]` 读取的属性：
- `Name` - 列名称
- `Type` - 列类型
- `Length` - 长度
- `Precision` - 精度
- `Comment` - 注释
- `IsPrimaryKey` - 是否主键
- `IsAutoIncrement` - 是否自增长

## 变更内容

### 1. 属性元数据方式变更

**变更**: `[MoQuery]` 不支持与 `[MoColumnAttribute]` 搭配工作

**原因**: 需求补充文档明确说明"MoQueryAttribute 不支持与 MoColumnAttribute 搭配工作"

### 2. 属性元数据处理策略调整

由于不能使用 `[MoColumn]`，需要为查询对象属性提供默认值策略：

| 属性 | MoTable (带 [MoColumn]) | MoQuery (无 [MoColumn]) |
|-----|------------------------|------------------------|
| Name | 从 `[MoColumn].Name` 读取 | 使用属性名称 |
| Type | 从 `[MoColumn].Type` 读取 | 根据属性类型推断 |
| Length | 从 `[MoColumn].Length` 读取 | 根据类型使用默认值 |
| Precision | 从 `[MoColumn].Precision` 读取 | 0 |
| Comment | 从 `[MoColumn].Comment` 读取 | 空字符串 |
| IsPrimaryKey | 从 `[MoColumn].Key` 读取 | false |
| IsAutoIncrement | 从 `[MoColumn].AutoIncrement` 读取 | false |

### 3. 源生成器逻辑调整

**变更**:
- 修改 `GetPropertyMetadata` 方法，为没有 `[MoColumn]` 的属性提供默认值
- 区分 `[MoTable]` 和 `[MoQuery]` 的属性处理逻辑
- 为 `[MoQuery]` 生成的属性使用自动推断的元数据

### 4. 使用示例更新

**变更**: 移除 `[MoColumn]` 特性

```csharp
// 查询对象 - 正确用法（无 [MoColumn]）
[MoQuery]
public partial class UserQuery
{
    public string? Username { get; set; }
    public int? MinAge { get; set; }
    public int? MaxAge { get; set; }
}
```

### 5. 接口适配

由于查询对象不需要 `[MoColumn]`，但 `IEntityModelProperty` 接口仍然需要这些属性值，需要：
- 使用属性名称作为 `Name`
- 根据属性类型推断 `Type`
- 使用合理的默认值

## 变更原因

需求补充文档新增明确约束：
- "MoQueryAttribute 不支持与 MoColumnAttribute 搭配工作"

这意味着：
1. 查询对象的设计理念与实体表不同，不需要复杂的元数据定义
2. 查询对象的属性元数据应该自动推断，而不是手动指定
3. 简化查询对象的定义，减少开发者的配置负担

## 可能产生的影响

### 正面影响

1. **简化配置**: 查询对象不需要手动指定每个属性的元数据
2. **减少代码**: 移除大量 `[MoColumn]` 特性配置
3. **自动推断**: 根据属性类型自动推断元数据，降低出错可能

### 潜在风险

1. **类型推断限制**: 某些类型可能无法精确推断为预期的 `ColumnType`
2. **自定义缺失**: 无法为特殊属性指定自定义元数据（如显示名称、校验规则）
3. **接口适配**: 需要为 `IEntityModelProperty` 提供合理的默认值

### 需要额外注意

1. 类型推断的准确性需要仔细测试
2. 可能需要新增 `[MoQueryProperty]` 特性用于需要自定义元数据的属性
3. 默认值策略需要与现有代码兼容
