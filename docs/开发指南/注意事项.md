# 开发注意事项

本文档记录在开发过程中积累的通用规范和注意事项，供后续开发参考。

## 功能文档结构规范

### 1. 文档结构要求

功能文档应包含以下章节，保持结构一致性：

```markdown
# X.X 功能名称

## 任务信息
- 任务号
- 任务名称
- 责任人
- 任务类型
- 任务描述

## 功能概述
- 功能描述
- 重要约束（如适用）

## 技术背景
### 现有实现
- 现有功能的详细说明
- 可使用表格对比
- 包含现有接口/类的定义

### 新增需求
- 新增功能的详细说明
- 与现有实现的对比

### 现有接口（如适用）
- 相关接口的完整定义
- 便于开发人员快速理解上下文

## 实现方案
### 1. 具体步骤
### 2. TODO: 待确认项
- 集中列出所有待确认项
- 每项包含选项和建议

## 使用示例
- 完整的代码示例
- 覆盖主要使用场景

## 实现步骤
- 开发任务清单
- 使用 checkbox 格式便于跟踪

## 参考文档
- 相关代码文件的相对路径链接

## 验收标准
- 明确的验收条件
```

### 2. 接口定义展示

对于涉及接口的功能，应在「技术背景」章节中提供完整的接口定义，包括：

- 所有公共方法
- 所有公共属性
- 完整的 XML 文档注释

### 3. TODO 项规范

- 所有 TODO 项集中到一个章节
- 每项包含选项和建议的默认值
- 格式：`TODO: 确认XXX（建议：XXX）`

### 4. 重要约束标注

在「功能概述」章节的顶部使用 `重要约束` 小节突出关键限制条件：

```markdown
## 功能概述

功能描述...

**重要约束**：
- 约束条件 1
- 约束条件 2
```

## 接口设计规范

### 1. 接口定义完整性

所有新增接口必须包含完整的公共成员定义：

```csharp
/// <summary>
/// 接口简述
/// </summary>
public interface IExample
{
    /// <summary>
    /// 属性描述
    /// </summary>
    int Count { get; }

    /// <summary>
    /// 方法描述
    /// </summary>
    /// <param name="paramName">参数描述</param>
    /// <returns>返回值描述</returns>
    void DoSomething(string paramName);

    /// <summary>
    /// 方法描述
    /// </summary>
    /// <typeparam name="T">类型参数描述</typeparam>
    /// <returns>返回值描述</returns>
    T? GetItem<T>();
}
```

### 2. AOT 编译友好性

- 避免使用 `System.Type` 类型
- 使用泛型或字符串名称代替
- 不使用反射相关代码

### 3. 接口与实现分离

- 先定义接口，明确契约
- 再提供默认实现（如 `DefaultXxx`）
- 静态访问点类（如 `XxxFactory`）委托给内部实现

### 4. 命名规范

- 接口以 `I` 开头：`IEntityModelSet`
- 默认实现以 `Default` 开头：`DefaultEntityModelFactory`
- 特性以 `Attribute` 结尾：`MoSetAttribute`
- 静态访问点使用简单名称：`ModelFactory`

### 5. 不可变集合规范

根据集合是否有数量概念，选择不同的返回类型：

- **有数量概念**：使用 `IReadOnlyList<T>`
  - 例如：`IEntityModelSet` 有 `Count` 属性，`GetModels()` 返回 `IReadOnlyList<IEntityModel>`
- **无数量概念**：使用 `IEnumerable<T>`
  - 例如：`IEntityModelFactory.GetAllModels()` 只需遍历，返回 `IEnumerable<IEntityModel>`

### 6. 模型唯一性规范

实体模型的唯一标识使用类全称，而非表名或查询对象名称：

- **类全称格式**：`{Namespace}.{ClassName}`
- **用途**：作为字典存储键、查询参数等
- **原因**：表名可能重复或变更，类全称更稳定且唯一

**API 设计示例**：
```csharp
// 主要查询方式 - 按类全称
IEntityModel? GetModel(string fullName);  // 例如："MyApp.Models.Order"

// 辅助查询方式 - 按表名
IEntityModel? GetModelByTableName(string tableName);  // 例如："orders"
```

### 7. 静态工具命名规范

所有静态工具类定义应该使用 `Utils` 结尾：

- **命名格式**：`{Domain}Utils`
- **示例**：`ModelUtils`、`StringUtils`、`DateUtils`
- **作用**：提供静态辅助方法，不涉及状态管理
- **对比**：涉及状态管理的类使用完整名称，如 `DefaultEntityModelFactory`

**API 设计示例**：
```csharp
// 静态工具类 - 使用 Utils 结尾
public static class ModelUtils
{
    public static string GetFullName(IEntityModel model);
}

// 工厂类 - 使用完整名称
public sealed class DefaultEntityModelFactory : IEntityModelFactory
{
    public IEntityModel? GetModel(string fullName);
    public void Add(IEntityModel model);
}
```

### 8. 源生成器独立规范

新增源生成功能时，应创建独立的源生成器，不修改现有源生成器：

- **原因**：保持现有源生成器的稳定性，降低引入风险
- **命名格式**：`{Domain}SourceGenerator`，如 `ModelSetSourceGenerator`
- **命名空间**：放在 `Delly.Modeling.Generator` 命名空间下
- **职责分离**：每个源生成器专注于一个特定的特性或功能

**对比示例**：
```csharp
// 新增功能 - 创建独立源生成器
[Generator]
public class ModelSetSourceGenerator : ISourceGenerator
{
    // 专门处理 [MoSet] 特性
}

// 参考现有实现 - 不修改
public class ModelTableSourceGenerator : ISourceGenerator
{
    // 专注于 [MoTable] 和 [MoQuery] 特性，保持不变
}
```

### 9. 源生成泛型 AOT 友好性规范

在源生成场景下，泛型方法的实现应避免运行时反射，使用编译时确定的类型信息：

- **原因**：LINQ 的 `OfType<T>()` 依赖运行时类型检查，不利于 AOT 编译优化
- **解决方案**：生成静态 `Type` 字段 + `switch case` 模式匹配
- **关键点**：`Type` 值在编译时确定，`switch` 编译器可优化为静态跳转表

**对比示例**：
```csharp
// ❌ 不推荐 - 依赖 LINQ 运行时类型检查
public IReadOnlyList<T> GetModels<T>() where T : IEntityModel
{
    return _models.OfType<T>().ToArray();
}

// ✅ 推荐 - 使用 typeof + switch case
private static readonly Type _userType = typeof(User);
private static readonly Type _orderType = typeof(Order);

public IEntityModel GetModel<T>() where T : class
{
    return typeof(T) switch
    {
        var t when t == _userType => User.GetEntityModel(),
        var t when t == _orderType => Order.GetEntityModel(),
        _ => throw new NotSupportedException($"不支持的模型类型: {typeof(T).FullName}")
    };
}
```

### 10. 源生成阶段 Type 属性 AOT 友好性规范

在源生成阶段，可以使用 `typeof(T)` 为生成的类添加 `Type` 类型的属性：

- **原因**：`typeof(T)` 在编译时确定，生成的代码为静态常量表达式，不影响运行时 AOT 编译
- **适用场景**：源生成器为模型类提供类型信息属性
- **命名建议**：使用 `ClassType` 或 `ModelType` 避免与关键字 `Type` 冲突

**代码示例**：
```csharp
// 生成的类中
public override Type ClassType => typeof(Order);
```

**注意事项**：
- 此规范仅适用于源生成器生成的代码
- 手动编写的代码仍应避免运行时使用 `System.Type`

## 代码规范

### 1. 公共成员文档注释

所有公共类、方法、属性、接口必须使用 `///` 格式的 XML 文档注释：

```csharp
/// <summary>
/// 公共方法描述
/// </summary>
/// <param name="paramName">参数描述，需与实际参数名匹配</param>
/// <returns>返回值描述</returns>
public void MyMethod(string paramName)
{
}
```

### 2. 私有成员注释

私有成员使用 `//` 格式注释，保留一个空格：

```csharp
// 私有成员描述
private readonly Dictionary<Type, IModel> _models;
```

### 3. 可空类型标注

接口和公共 API 中的可空对象参数使用 `object?` 格式。

## .NET 版本兼容性规范

### 1. 接口定义可空类型约束

项目目标框架为 `netstandard2.0` 和 `net6.0`，`.NET Standard 2.0` 不支持 `Nullable` 引用类型。

**接口定义原则**：
- 所有接口中的函数参数和返回值应使用非空类型 `object`
- 不使用可空类型 `object?`
- 实现类内部可以检查 null 并返回 null

**接口定义示例**：

```csharp
// ✅ 正确 - 使用非空类型
public interface IParsable
{
    static abstract object TryParse(object obj);
}

public interface IBaseModel
{
    object Parse(object obj);
    object TryParse(object obj);
}

// ❌ 错误 - 使用可空类型（不兼容 .NET Standard 2.0）
public interface IParsable
{
    static abstract object? TryParse(object? obj);
}
```

**实现类示例**：

```csharp
// 实现类中可以返回 null
public object TryParse(object obj)
{
    if (obj == null)
        return null;  // 允许返回 null

    if (obj is int value)
        return value;

    return null;  // 解析失败返回 null
}
```

**注意**：此规范为项目默认约束，所有函数定义时都需要默认考虑不同 .NET 版本的兼容性。

### 2. 条件编译使用规范

当需要在多个目标框架上支持不同的语言特性时，使用条件编译指令。

**条件编译原则**：
- 使用 `#if !NETSTANDARD2_0` 区分 .NET Standard 2.0 和其他框架
- 在 .NET 6.0 及以上使用 `#nullable enable` 启用可空引用类型
- 不同框架使用不同的接口定义和实现方式

**条件编译示例**：

```csharp
#if !NETSTANDARD2_0
#nullable enable
#endif

public interface IParsable
{
#if !NETSTANDARD2_0
    static abstract object? TryParse(object? obj);
#else
    static abstract object TryParse(object obj);
#endif
}
```

**源生成器中的条件编译**：

```csharp
// 生成的代码示例
#if !NETSTANDARD2_0
public override object? TryParse(object? obj)
#else
public override object TryParse(object obj)
#endif
{
#if !NETSTANDARD2_0
    if (typeof({ClassName}).IsAssignableTo(typeof(IParsable)))
    {
        return {ClassName}.TryParse(obj);
    }
    throw new ArgumentException($"Type {ClassName} does not implement IParsable interface");
#else
    // .NET Standard 2.0 不支持静态接口成员
    throw new NotImplementedException("IParsable is not supported in .NET Standard 2.0");
#endif
}
```

**注意事项**：
- 源生成器需要为不同框架生成不同的代码块
- 每个框架都需要单独的测试覆盖
- 条件编译会增加代码维护复杂度，谨慎使用

### 3. 静态接口成员约束

静态接口成员（`static abstract`）是 .NET 7+ 引入的特性，项目目标框架 `netstandard2.0` 和 `net6.0` 均不支持。

### 10. 禁止反射约束

本项目的源生成器和运行时代码均禁止使用反射，以确保 AOT 编译友好性和运行时性能。

**禁止内容**：
- 不使用 `System.Reflection` 命名空间下的类型和方法
- 不使用 `typeof(T).GetMethod()`, `Invoke()` 等反射调用
- 不使用动态类型查找和方法分发

**替代方案**：
- 源生成阶段：硬编码调用，使用 `switch case` 模式匹配
- 运行时阶段：使用字典 + 委托，或接口模式

**示例对比**：
```csharp
// ❌ 错误 - 使用反射
var method = typeof(MyType).GetMethod("TryParse");
var result = method.Invoke(null, new object[] { obj });

// ✅ 正确 - 硬编码调用
if (typeof(MyType).IsAssignableTo(typeof(IParsable)))
{
    return MyType.TryParse(obj);
}
```
